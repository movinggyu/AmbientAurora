# AmbientAurora 프레임드랍 원인 분석 및 해결 방향

## 개요

이 문서는 현재 프로젝트에서 프레임드랍이 발생할 가능성이 있는 지점을 정리하고, 각 원인별로 어떤 방식으로 개선할 수 있는지 기록한 문서입니다.

현재 구조에서 프레임드랍은 주로 다음 두 축의 결합으로 발생할 가능성이 큽니다.

- 렌더 루프가 과도하게 반복되거나, 프레임이 늦을 때 다음 프레임이 밀리는 구조
- 풀스크린 OpenGL 렌더링과 QML UI 애니메이션이 동시에 동작하면서 합성 비용이 증가하는 구조

---

## 1. 가장 가능성이 높은 원인

### 1.1 OpenGL 위젯의 렌더 루프가 16ms 타이머에 직접 의존함

위치:
- [src/ui/OpenGLWidget.cpp](src/ui/OpenGLWidget.cpp)

현재 방식은 `QTimer`를 이용해 약 16ms마다 `update()`를 호출하고, 그에 따라 `paintGL()`이 실행됩니다.

문제점:
- 실제 프레임이 끝나기 전에 다음 프레임 요청이 들어갈 수 있음
- 한 프레임이 늦어지면 다음 프레임도 함께 밀려서 시각적으로 흔들리는 현상이 생김
- 노트북과 데스크탑의 성능 차이 때문에 동일한 코드에서도 드랍 패턴이 달라질 수 있음

해결 방향:
- `QTimer` 기반 대신 `QOpenGLWidget`의 자연스러운 프레임 업데이트 방식과 `QElapsedTimer` 기반으로 프레임 간격을 제어
- 프레임 시간 측정 후, 초과한 경우에는 강제로 여러 번 그리지 않도록 제한
- 가능하면 `update()` 대신 `QWindow::requestUpdate()` 또는 `QTimer`의 간격을 유연하게 조절하는 방식으로 전환

---

### 1.2 OpenGL 위젯과 QML 위젯을 동시에 겹쳐서 렌더링함

위치:
- [src/ui/MainWindow.cpp](src/ui/MainWindow.cpp)
- [src/ui/OpenGLWidget.cpp](src/ui/OpenGLWidget.cpp)

현재 앱은 배경에 OpenGL 위젯을 두고, 그 위에 QML 레이어를 올려서 겹쳐 그립니다.

문제점:
- 두 렌더러가 독립적으로 동작하므로 합성 비용이 증가함
- 투명 배경과 중첩 구조에서 윈도우 시스템의 compositor와 충돌할 수 있음
- 특히 Windows 환경에서는 레이어 합성이 더 민감하게 나타날 수 있음

해결 방향:
- 가능하면 QML과 OpenGL을 하나의 렌더 경로로 통합
- 최소한 QML 레이어의 렌더링 수를 줄이고, 필요한 경우에는 OpenGL 위젯 내부에 직접 렌더링하는 방식으로 변경
- 불필요한 투명/중첩 레이어를 제거하여 합성 부담을 줄임

---

### 1.3 QML이 OpenGL 경로로 강제되어 있음

위치:
- [src/mainTest.cpp](src/mainTest.cpp)

코드는 `QQuickWindow::setGraphicsApi(QSGRendererInterface::OpenGL)`를 사용합니다.

문제점:
- Qt Quick의 렌더링 경로가 OpenGL로 고정되면, 플랫폼/드라이버/Qt 버전에 따라 성능이 불안정해질 수 있음
- OpenGL 위젯과 QML 렌더러가 같은 스레드/컨텍스트 환경에서 함께 동작하면 병목이 생기기 쉬움

해결 방향:
- 먼저 OpenGL 강제 설정을 제거하고 기본 렌더링 경로를 확인
- 필요 시 Qt Quick의 권장 경로에 맞춰 렌더러를 조정
- 가능하면 QML UI의 애니메이션은 단순화하고, 핵심 시각 효과는 OpenGL 쪽으로 우선 처리

---

## 2. GPU 비용이 큰 요소

### 2.1 풀스크린 셰이더가 매 프레임 매우 무거움

위치:
- [src/render/ScreenRenderer.cpp](src/render/ScreenRenderer.cpp)

현재 프래그먼트 셰이더는 다음과 같은 연산을 매 픽셀마다 수행합니다.

- `pow`
- `sin` / `cos`
- `atan`
- `mod`
- `smoothstep`
- 색 공간 변환 함수들

문제점:
- 전체 화면에 대해 매 프레임 실행되므로 GPU 부담이 큼
- 특히 화면 해상도와 애니메이션 속도가 높아질수록 비용이 급격히 커짐
- 노트북에서는 이 비용이 더 크게 느껴지고, 데스크탑에서도 드랍이 관찰될 수 있음

해결 방향:
- 셰이더 연산을 단순화하고, 불필요한 색 공간 변환을 줄임
- `u_Progress`에 따라 연산량이 변하는 방식으로 최적화
- 고주파 연산은 텍스처 기반 혹은 CPU 미리 계산된 데이터로 대체 가능
- 필요 시 해상도에 따라 셰이더 강도를 조정

---

### 2.2 화면을 매 프레임 지우고 다시 그리며 비용이 누적됨

위치:
- [src/render/ScreenRenderer.cpp](src/render/ScreenRenderer.cpp)

현재 구현은 `glClear()`와 전체 화면 드로우를 매 프레임 수행합니다.

문제점:
- 단일 사각형이라 큰 문제는 아니지만, 앞의 셰이더/합성 비용과 결합되면 병목이 됨
- `paintGL()`에서 한 번, QML에서 또 한 번 렌더링을 유발하는 구조라 비효율적임

해결 방향:
- 불필요한 clear와 상태 변경을 줄임
- 상태가 바뀌지 않는 경우에는 재사용 가능한 렌더 경로로 정리
- 가능한 경우 배경을 사전 렌더링하거나 프레임 단위 갱신 빈도를 낮춤

---

## 3. UI 애니메이션이 프레임드랍을 증폭시킬 수 있음

### 3.1 QML의 opacity/translate 애니메이션

위치:
- [src/ui/qml/BottomBar.qml](src/ui/qml/BottomBar.qml)
- [src/ui/qml/GlassPanel.qml](src/ui/qml/GlassPanel.qml)
- [src/ui/qml/LogoButton.qml](src/ui/qml/LogoButton.qml)

현재 UI는 `NumberAnimation`과 `Behavior on`를 사용해 투명도와 위치를 애니메이션합니다.

문제점:
- 여러 요소가 동시에 애니메이션될 때 렌더링 부담이 증가함
- 반투명 배경과 효과가 겹치면 compositor 부담이 커짐
- Windows 환경에서 특히 더 눈에 띔

해결 방향:
- 동시에 애니메이팅되는 요소 수를 줄임
- 애니메이션 지속 시간과 복잡도를 낮춤
- 필요하지 않은 `MultiEffect`나 투명 레이어 사용을 줄임

---

### 3.2 `MultiEffect` 사용

위치:
- [src/ui/qml/LogoButton.qml](src/ui/qml/LogoButton.qml)

`MultiEffect`는 시각 효과를 주지만, GPU 리소스와 합성 비용을 증가시킬 수 있습니다.

문제점:
- 단순한 이미지 표시보다 더 많은 픽셀 연산이 필요함
- 반투명 UI와 결합되면 드랍이 더 잘 보임

해결 방향:
- 단순 이미지 렌더링으로 대체 가능하다면 우선 제거
- 효과가 꼭 필요하다면 낮은 해상도나 제한된 조건에서만 사용

---

## 4. 디버깅 관점에서 확인하면 좋은 포인트

다음 항목을 조합하면 원인 파악이 훨씬 쉬워집니다.

1. `paintGL()` 진입/종료 시간 측정
2. 프레임별 소요 시간 로그 기록
3. QML 애니메이션을 일시적으로 비활성화했을 때의 차이 확인
4. 셰이더를 단순한 색 채우기 코드로 바꿔서 성능 차이 비교
5. 전체 화면 해상도와 창 크기 변화 시 성능 차이 확인

---

## 5. 권장 개선 순서

### 우선순위 1: 구조 개선
- OpenGL 렌더 루프와 QML 렌더 경로 분리 여부 검토
- 중첩 렌더러 구조를 단순화
- QML과 OpenGL이 동시에 과도하게 그리는 구조를 줄임

### 우선순위 2: 셰이더 최적화
- 불필요한 연산 제거
- 색 공간 변환 로직 간소화
- 고비용 함수 사용 줄이기

### 우선순위 3: UI 애니메이션 단순화
- `NumberAnimation` 수를 줄임
- `MultiEffect`와 투명 레이어를 최소화
- 필요 없는 연속 애니메이션 제거

### 우선순위 4: 프로파일링
- 실제 측정값을 바탕으로 병목 지점을 확인
- 프레임 시간 분포를 시각화하여 어떤 구간에서 드랍이 나는지 확인

---

## 6. 결론

현재 프로젝트에서 프레임드랍의 핵심 원인은 다음으로 정리할 수 있습니다.

- OpenGL과 QML이 동시에 겹쳐 렌더링되는 구조
- 매 프레임 전체 화면 셰이더 연산이 무거운 구조
- QML 애니메이션과 효과가 합성 비용을 증폭시키는 구조

이 세 가지가 결합되면, 노트북에서는 더 쉽게 체감되고 데스크탑에서도 특정 조건에서 드랍이 생기는 패턴이 나타날 수 있습니다.
